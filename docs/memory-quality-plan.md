# 记忆碎片质量优化方案

## 现状诊断

### 数据（2026-05-31，198 个碎片）

| type | 数量 | 占比 | 初始热度 |
|------|------|------|----------|
| fact | 184 | 93% | **100** ← 最高 |
| reflection | 10 | 5% | 70 |
| preference | 3 | 1.5% | 90 |
| event | 1 | 0.5% | 80 |

热度分布：133 个碎片热度 = 100（67%），完全没有区分度。

### 根因

**1. 初始热度倒挂**

fact 给 100 最高分，reflection 给 70 最低分。fact 是最没信息量的类型（"我在吃饭"、"我在写代码"），reflection 才是对 toge 有洞察的（"我觉得自己拖延是因为害怕做不好"）。当前热度设计让噪音压过了信号。

**2. 分类几乎不存在**

- `extractFromTurn()`：正则匹配后，剩余句子**全部标 fact**，不判断内容
- `extractFromDiary()`：只有 7 个关键词判断 reflection / preference，其余全标 fact
- fact 正则 `/我(?:在|的|有|是|住|去|去了)\S+/g` 匹配几乎所有带"我"的句子

**3. 没有质量门控**

以下内容全部入库了：
- 标题行："回顾一下这十几个小时都折腾了什么："
- 列表项："**聊天页**：有了小螃蟹桌宠..."
- 纯猜测："可能睡着了或者在专注写代码"
- 过渡句："从 ADB 方案聊到 MacroDroid webhook..."

**4. 去重逻辑粗糙**

`f.content.includes(trimmed) || trimmed.includes(f.content)` —— 短字符串包含匹配会导致"我在吃饭"阻止"我在食堂吃饭因为不想点外卖"入库。

---

## 改进方案

### 一、初始热度重设

```
// 当前
fact: 100, event: 80, reflection: 70, preference: 90

// 改为
reflection: 85,  // 最有价值：toge 对自己的洞察
preference: 80,  // 重要：喜欢什么讨厌什么
event: 65,       // 有用：做了什么决定、发生了什么
fact: 40,        // 基础信息，初始不突出
```

理由：让真正有区分度的碎片从起点就拉开差距。fact 热度低不代表没用——被频繁召回时 `touch(+5)` 和 `boostHeat(+3)` 会自然抬升。但"我在吃饭"这种一辈子没人搜的 fact 就该沉底。

### 二、句子级智能分类

**extractFromTurn()** 的句子分割后，不再全标 fact，而是过一遍分类：

```
分类规则优先级从高到低：
1. 含「觉得/感觉/发现/意识到/原来/明白/好像/可能是我」→ reflection
2. 含「喜欢/讨厌/爱/恨/想/希望/想成为/想变成」→ preference
3. 含「决定/打算/计划/要开始/放弃/不…了/从今天起」→ event
4. 含「在/有/是/住/去/去了/来自」→ fact
5. 以上都不匹配 → 检查是否有实质信息量，没有就跳过不存
```

### 三、质量门控

入库前增加以下过滤：

| 过滤条件 | 示例（会被丢弃） |
|---------|----------------|
| 长度 < 6 字符 | "我在写代码" |
| 纯标题/列表项 | "回顾一下这十几个小时："、"**聊天页**：…" |
| 纯猜测且无结论 | "可能睡着了"、"可能在忙" |
| 纯时间线叙述 | "从 A 聊到 B"（无情感/决策内容） |
| Markdown 标记残留 | 含 `**`、`# `、`- ` 开头的列表项 |

### 四、修复 fact 正则

```diff
- /我(?:在|的|有|是|住|去|去了)\S+/g    ← \S+ 在中文无空格时吞到行尾
+ /我(?:在|的|有|是|住|去|去了).{1,30}/g  ← 限制匹配长度，且只匹配有实质内容的
```

### 五、去重改为精确匹配

```diff
- f.content.includes(trimmed) || trimmed.includes(f.content)
+ f.content === trimmed
```

当前的同日去重本身是按天的，跨天重复是允许的（同一条信息在不同天被提起也有意义）。

---

## 预期效果

- type 分布：fact 占比从 93% 降到 ~50%，reflection + preference 升到 ~30%
- 热度分布：呈现 20-100 的梯度，高热度碎片真正有信息量
- 碎片总量减少 ~30%（质量门控过滤噪音）
- `injectMemoryContext()` 召回的质量提升：搜出来的是 toge 说过的重要的话，不是"在吃饭"

---

## 待讨论

1. **fact 要不要完全取消？** 有人觉得"toge 在杭州萧山区"、"toge 有 ADHD"这类事实也重要。方案里保留但降初始热度，靠被召回抬升。另一个思路是把 fact 拆成 `identity`（身份事实）和 `trivial`（日常琐事），后者不存。

2. **要不要引入 LLM 做提取？** 当前是纯规则。LLM 提取质量会高很多（能理解上下文、能判断重要性），但每次对话后调一次 API 有成本和延迟。可以先优化规则，效果不够再加 LLM。

3. **旧碎片怎么处理？** 改完后新碎片按新规则，旧的 184 个 fact 要不要批量清理/重分类？还是等自然衰减（每天 -1，100→10 要 90 天）？
