# 迭代日志书写指南

> 这是 [iteration-log.md](iteration-log.md) 的配套规范文档，本身不含日期条目。

> **迭代日志 ≠ git log。** git 管"改了什么文件"；迭代日志管 git 做不到的——**为什么这样改、怎么发现的、当时怎么想的。**

## 模板

```markdown
## YYYY-MM-DD · 一句话概括 · `#tag1` `#tag2`

### 背景

（1-3 句）

### 关键踩坑

| 坑  | 现象 | 根因 | 解  |
| --- | ---- | ---- | --- |

### 设计决策（可选）

（选 A 不选 B 的理由）

> 📦 [`abc1234`](commit-url) 这组 commit 干了什么
> 关键文件：xxx.js、yyy.js
```

## 规则

1. **严格倒序，插到正确位置** — 新条目按日期插入，最新的在最上面。**不要追加到文件末尾。** 不确定位置就看日期标题，06-30 在 06-29 上面，依此类推
2. **标题带标签** — 1-3 个简短英文标签（如 `#turn-gate`），方便以后 `grep` 搜同类踩坑；尽量复用已有标签（grep 现有标题看有没有类似的），没有就新建。不维护单独的索引——标签直接搜标题就够，省得两处同步又漏更新
3. **改动用 commit 链接，不写文件表** — commit 完把 hash 贴过来，精确 diff 点链接看
4. **踩坑表是核心** — 现象 → 根因 → 解，三列足够
5. **叙事有节制** — 过程超过 5 行的挪单独文档，这里留一句话 + 链接

## 什么不写

- ❌ 手写改动文件表 → 归 git
- ❌ 功能效果清单 → 归功能文档
- ❌ 超过 5 行的过程叙事 → 挪单独文档
- ❌ 还没确认的猜测

## 归档

每月初把上个月条目移到 `iteration-log-YYYY-MM.md`，只保留当月 + 上月活跃。归档时每条压缩为标题 + 踩坑表 + commit 链接。
